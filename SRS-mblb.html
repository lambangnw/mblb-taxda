01

Arsitektur RBAC & Security (Role-Based Access)

8 Entitas Actor, Dynamic Sidebar Render, Route Guards `checkAuth()`

Definisi Aktor (Hierarchy)

1. Admin Sistem — Superuser. Mengelola CRUD Master Harga Mineral, Kode Rekening, Tutup Buku Tahunan, dan Konfigurasi Menu Sidebar (Custom RBAC).
2. Kepala Dinas — Eksekutif (Read-Only). Memiliki akses ke Dashboard Laporan Realisasi Granular (Harian/Bulanan).
3. Kabid Penetapan — Managerial. Melakukan ACC (Approval) terhadap Draft SPTPD sebelum sah diterbitkan dan disetujui untuk menjadi piutang.
4. Admin Penetapan — Operator Bapenda. Menerima laporan harian lapangan, me-generate Draft SPTPD Wajib Pajak (Assisting WP), dan menerbitkan Surat Teguran.
5. Staff Lapangan — Mobile Data Entry. Datang ke lokasi tambang (quarry) mencatat volume (ritase/tonase) ekskavasi material MBLB per hari.
6. Bidang Pendaftaran dan Pendataan — Loket Pelayanan. Mendaftarkan NPWPD baru, pendaftaran berkas objek pajak MBLB.
7. Bidang Penagihan — Keuangan. Verifikasi Setoran (BKP/Kas Daerah). Mencetak SSPD Lunas, memonitor piutang / aging tunggakan.
8. Wajib Pajak — Publik. Pemilik tambang. Dapat login untuk melihat Tagihan (SPTPD Rampung yang divalidasi dinas), riwayat setor, dan SSPD.

⚠️ IMPLEMENTASI ROUTE GUARDS & SIDEBAR

DEV-1.1

Backend/Sistem harus memakai **Single Source of Truth** untuk session: `localStorage.getItem('mblb_current_user')` melalui fungsi helper `getCurrentUser()`. Jangan ada hardcode variabel `mblb_user`.

DEV-1.2

Dynamic Sidebar: Sidebar menu TIDAK BOLEH di-hardcode di HTML. Semua element `.sidebar-menu` harus dikosongkan secara statis dan dirender otomatis via `renderDynamicSidebar()` di `shared.js` menggunakan relasi `MBLB_MENU_MASTER` dan `MBLB_DEFAULT_RBAC`.

DEV-1.3

Seluruh folder internal (/Admin Sistem, /Kepala Dinas, dll) WAJIB dilindungi oleh pemanggilan `checkAuth()` di fase `DOMContentLoaded`. User dengan Role A yang nekat mengakses index HTML milik Role B harus otomatis ter-kick / dialihkan ke halamannya sendiri.

Siklus Bisnis Inti (Input & Output Tiap Fase)

Menganut "Self-Assessment Berbantu (Assisted)". SPTPD dibuatkan oleh dinas berdasarkan validasi riil harian, lalu disetujui WP.

1 Fase: Registrasi Wajib Pajak Baru

Aksi / Input

  • WP datang ke Bidang Pendaftaran dan Pendataan membawa KTP & Izin Tambang.
  • Petugas meng-input form pendaftaran WP di sistem (Data Diri, Alamat Tambang, Titik Koordinat).

HASIL / OUTPUT

  • NPWPD (Nomor Pokok Wajib Pajak Daerah) unik per WP di-generate otomatis.
  • Akun Login Web WP: Sistem mengirimkan password acak / standard ke email/SMS WP untuk akses portal `Wajib Pajak`.
2 Fase: Deklarasi Awal SPTPD

Aksi / Input

  • WP mendeklarasikan "Rencana Produksi" (contoh: akan menambang Pasir Sungai sebanyak 10.000m3 bulan ini).

HASIL / OUTPUT

  • Database menetapkan Target Kuota / Rencana untuk quarry WP tersebut yang akan diawasi secara harian.
3 Fase: Pencatatan Produksi Aktual (Harian)

Aksi / Input

  • Staff Lapangan menerima penugasan langsung dari Kabid.
  • Melakukan inspeksi ke lokasi galian C WP dan mencatat volume (ritase/tonase) ekskavasi material MBLB.
  • Input data via Mobile UI yang telah dimodernisasi (Glassmorphism Indigo) untuk efisiensi tinggi.

HASIL / OUTPUT

  • Data Transaksi Laporan Harian tersimpan di database dengan status `DRAFT_LAPORAN`.
  • Update Real-time pada Dashboard Eksekutif Kepala Dinas (Grafik Agregat Volume Tambang Kabupaten).
4 Fase: Rekapitulasi Bulanan

Aksi / Input

  • Sistem secara otomatis mengumpulkan seluruh Laporan Harian (Ritase/Volume) dari Staff Lapangan selama 1 bulan penuh per WP.
  • Sistem menghitung total akumulasi gross material yang digali di lapangan (m3 / tonase).

HASIL / OUTPUT

  • Draft Rekapitulasi Bulanan: Tabel akumulatif riwayat produksi harian sebulan penuh. Inilah pondasi data mentah yang jadi rujukan pembuatan Draft SPTPD bulanan ke depannya.
5 Fase: Verifikasi & Penerbitan SPTPD (Cut-off)

Aksi / Input

  • Admin Penetapan di kantor Bapenda melakukan generate/tutup buku bulanan pada tanggal 28-31 tiap bulan.
  • Admin me-review kompilasi laporan harian Aktual Laporan Lapangan.
  • Admin Penetapan me-generate Draft SPTPD sebagai bentuk fasilitasi (Assisted) kepada WP.
  • Kabid Penetapan me-review dan klik "Setujui SPTPD".

HASIL / OUTPUT

  • Draft SPTPD (Internal): Kalkulasi kotor pajak MBLB. Status naik ke `VERIFIKASI_NOTA`.
  • Dokumen SPTPD Rampung: Surat yang disahkan. Status naik ke `UNPAID_SPTPD` (Sah menjadi Tagihan Terbuka bagi WP).
  • Notifikasi / Tagihan Tampil: SPTPD langsung tampil di portal Wajib Pajak untuk dibayar. Jatuh tempo dihitung +30 hari sejak rilis.
6 Fase: Pembayaran & Pelunasan

Aksi / Input

  • WP (via sistem bank/portal) membayar tagihan SPTPD melalui Transfer VA / Kas Daerah.
  • Jika uang sudah masuk via Bank, Bidang Penagihan mencocokkan mutasi bank dengan nominal tagihan, lalu klik "Verifikasi Lunas".

HASIL / OUTPUT

  • Status Tagihan berubah dari UNPAID_SPTPD menjadi PAID_SSPD.
  • Dokumen SSPD (Setoran): Kwitansi Bukti Lunas tercetak. Ini adalah bukti bayar mutlak (resmi) yang bisa di-download oleh WP.
  • Realisasi PAD Naik: Pendapatan Asli Daerah (PAD) di Dashboard Kepala Dinas resmi diakui sebagai Realisasi.
03

Algoritma Kalkulasi Pajak, Opsen & Sanksi

Formulasi Matematika sesuai UU HKPD (Opsen MBLB 25%)

1. Dasar Pengenaan Pajak (DPP)

Nilai Jual Hasil Tambang. Dihitung per material.

DPP = Volume_Material × Harga_Dasar_Pasar

*Harga Pasar bersifat dinamis, diambil (di-snapshot) berdasarkan Master Harga per tanggal ditetapkannya SPTPD.

2. Perhitungan Pokok & Opsen

Pajak MBLB secara default adalah 20% menurut Perda. Dari nominal tersebut (Opsional), dikenakan Opsen Provinsi sebesar 25% dari Pokok.

Pajak_Kabupaten = DPP × Tarif_Dasar (20%)
Opsen_Provinsi = Pajak_Kabupaten × Tarif_Opsen_Provinsi (contoh: 25%)
Total_Tagihan_Pajak = Pajak_Kabupaten + Opsen_Provinsi
EC-3.1 SANKSI 2% PER BULAN

Jika Wajib Pajak membayar melewati Tanggal Jatuh Tempo SPTPD (biasanya 30 hari sejak terbit), sistem WAJIB menghitung denda keterlambatan secara progresif otomatis.

// Pseudocode untuk Billing Generation (SSPD Input)

Bulan_Telat = CEIL(DateDiff_Months(Tgl_Bayar, Tgl_Jatuh_Tempo))

IF Bulan_Telat < 0 THEN Bulan_Telat = 0

IF Bulan_Telat > 24 THEN Bulan_Telat = 24 // Max denda 24 bln

Nilai_Denda = Total_Tagihan_Pajak × (2%) × Bulan_Telat

Tagihan_Final_Dibayar = Total_Tagihan_Pajak + Nilai_Denda

⚠️ Sistem pembayaran (Bidang Penagihan) TIDAK BOLEH menerima setoran SSPD yang nominalnya kurang dari Tagihan_Final_Dibayar (termasuk dendanya).

04

Modul Surat Teguran & Report Analitik

Mekanisme punishment & laporan Eksekutif (Kadis)

Surat Teguran WP (SP 1, SP 2, SP 3)

Admin Penetapan berhak mendaftar WP yang melewati batas jatuh tempo (piutang SPTPD). Status surat teguran dapat di-track dan dicetak fisik.

DEV-4.1: Tabel `Teguran` berelasi 1-to-M dengan `Wajib Pajak`. Kolom `tingkat_teguran` = ENUM('SP 1', 'SP 2', 'SP 3').

DEV-4.2: Auto-suspend WP yang mendapati SP 3 (Pencabutan Izin sementara) — flag `is_active` WP menjadi `false` sampai denda dibayar.

Laporan Realisasi & Tutup Buku Tahunan

Dashboard Utama Kepala Dinas memfilter Penerimaan Pokok dan Sanksi Denda. Jika tahun anggaran habis, dilakukan "Tutup Buku Baru".

DEV-4.3: Aggregation Rule: SELECT SUM(pokok), SUM(denda) WHERE status = 'PAID_SSPD'. Jika tagihan masih `UNPAID_SPTPD`, nilai itu disebut PIUTANG, DILARANG masuk ke grafik REALISASI PENDAPATAN.

DEV-4.4: Tutup Buku: Freeze/Kunci seluruh database (SSPD & SPTPD) pada tanggal 31 Desember. Mencegah modifikasi/delete SSPD setelah di-audit oleh BPK.
